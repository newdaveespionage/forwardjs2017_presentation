Provides solutions for:

  2. Files needing to be changed before they can go to the client

Q: Wait, what's a module?

A: A system of writing js that allows files to declare dependencies on each other by importing and exporting modules from those files.

Webpack...

Q: Wait, Javascript has modules?

A: Yes. It's complicated.

Asynchronous Module Definition

			define({
				pluralize: function(count, word){
					return count === 1 ? word : word + 's';
				}
			});

CommonJS Modules

			function pluralize(count,word) {
				return count === 1 ? word : word + 's';
			};
			module.exports { pluralize:pluralize }

ES6 Modules

			export { pluralize }
			function pluralize(count,word) {
				return count === 1 ? word : word + 's';
			};

Webpack...

  1. Works with CommonJS modules
    (has an adapter for AMD, uses Babel for ES6 Modules)
  2. Works with single or multiple points of entry
  3. Can leverage plugins in its loader system to do things such as
    transpile, uglify, minify, supply polyfills
  4. Tree-shaking (but only if you're using explicit export definitions)

Summary: Webpack

If you want to:

  • Have control over how your front-end is loaded by the browser
  • Ensure your dependencies are resolved before your code goes live
  • Tie the parts of your build configuration together

Webpack is a comprehensive build tool that meets these needs

Redux -> Async

  • Out of the box, Redux is synchronous
  • Thunks provide a way to hide the async behavior
  • Sagas provide an entire framework for using ES6 generators
    to control "side-effects"

Thunks

  • Handle dispatch asyncronously for you when you return a function instead of a state
  • Allows functions that require special async considerations to look and act the same outside of their own code

Sagas

  • Uses ES6 generators which are functions that maintain their own state out of the box
  • Generators are inherently testable as standalone components because they respond without the parent framework
  • Multiple actions (repeated button pushing) are handled using takeEvery and takeLatest

Summary: Thunks vs. Sagas

  • Thunk code is more succint, and handles simple side-effects without adding a lot to the codebase
  • Sagas provide more robust support for a multitude of side-effect states, and are more testable
